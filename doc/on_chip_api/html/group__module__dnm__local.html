Description

Wrapper around the raw local interface.

Note
To enable you to modify its behavior, the source code of this application module is provided as part of the SmartMesh Firmware Development Kit.

Communication with the SmartMesh protocol stack happens through a Channel. The format of the requests, responses and notifications exchanged over this channel is documented in Raw Local Interface. To speed up the development of your application, this application-level module turns the channel interaction into simple function calls.

The functions in this module allow you to drive the mote's protocol stack:

  • have mote search for and join a network,
  • request services from the manager,
  • send to and receive data from the network,
  • etc.

This module has the following features:

  • It hides complexities of channel-based communication behind simple function calls, speeding up the development of your application.
  • The set of functions implement the complete interface you need to drive the SmartMesh protocol stack.
  • It uses a callback registration mechanism to notify your application about events from the protocol stack, such as receving a packet from the network.

It is possible to use the module in one of two different modes:

  • In PASSTHROUGH_OFF mode, the module parses received notifications and invokes registered callbacks for each one with proper parameters. This is the default way to use the module
  • In PASSTHROUGH_ON mode, the module invokes one of two possible callbacks (one for events, and the other for all other notifications) and leaves the parsing of notifications up to the user application. See dnm_loc_registerPassthroughEvNotifCallback() and dnm_loc_registerPassThroughNotifCallback(). This mode may be useful if the application requires further processing of the received packet (e.g. for queueing). The callbacks allow the application to swap buffers for the module's next channel receive operation.

To use this module, your application must follow these steps:

  • Create a synchronous channel and register it with the DN_MSG_TYPE_NET_NOTIF message type.
  • Initialize the module by calling dnm_loc_init().
  • Optionally, register callback functions for event and frame reception notifications by calling dnm_loc_registerEventNotifCallback() and dnm_loc_registerRxNotifCallback().
  • Continuously call dnm_loc_processNotifications(), which internally waits for the next notification and processes it depending on the command received.
  • Use the provided API calls to issue commands to the stack (e.g. dnm_loc_joinCmd() or dnm_loc_sendtoCmd()).
Note
The SmartMesh Firmware Development Kit comes with numerous example applications which do the above, and which are available in source code.
Warning
Call functions in this module only after calling dnm_loc_init(), and making sure that it returned a DN_ERR_NONE error code.

dn_error_t dnm_loc_getParameterCmd ( INT8U  paramId,
INT8U *  payload,
INT8U  txPayloadLen,
INT8U *  rxPayloadLen,
INT8U *  rc 
)

Get some configuration parameter.

Parameters
[in]paramIdIdentifier of the parameter to be retrieved.
[in,out]payloadPoints to both the structure to pass with the GET command, and contains the result after the function returns.
[in]txPayloadLenNumber of bytes in the structure passed with the GET commmand.
[out]rxPayloadLenLength of the result, i.e. number of bytes written back into the payload buffer.
[out]rcLocation to write the return code to (details below).

This function calls the DN_API_LOC_CMD_GETPARAM command of the local interface. There are in two elements which can be considered "return codes" when calling this function:

  • The value returned by this function merely indicates whether the command could be issued to the local interface.
  • The outcome of that call is written at the location pointed to by the rc parameter. Consult the getParameter section relative to paramId you are getting in the "SmartMesh IP Mote Serial API Guide" (http://www.linear.com/docs/41886); it lists the possible return codes and their meaning.
Postcondition
After the function returns successfully, the location of payload contains the following information:
  • byte 0: the value of the return code.
  • byte 1: the paramId.
  • byte 2 and more: the value of the requested parameter. You should therefore allocate two extra bytes to the payload buffer for the return code and paramId. Similarly, the value of rxPayloadLen should include those two bytes.
Note
The payload parameter points to a buffer you have previously allocated. You should allocate a buffer large enough to fit either the structure sent to the function (txPayloadLen bytes), or the data written by this function (rxPayloadLen bytes). That is, when allocating this buffer, it should be of size max(txPayloadLen,rxPayloadLen) bytes.
Returns
DN_ERR_NONE if the function completes successfully.
DN_ERR_ERROR if the function can not be completed successfully.

INT8U  numSubtests,
dn_api_loc_testrftx_subtestparam_t *  subTests,
INT8U *  rc 
)

Have the mote perform a radio transmit test.

Parameters
[in]typeTransmition type. Acceptable values are:
  • DN_API_RADIOTX_TYPE_CW for a continuous (unmodulated) wave.
  • DN_API_RADIOTX_TYPE_CM for a continuous modulated signal.
  • DN_API_RADIOTX_TYPE_PKT to send some number of packets.
  • DN_API_RADIOTX_TYPE_PKCCA to send some number of packets with CCA enabled.
[in]maskMask of channels (0-15) enabled for the test. Channel 0 (resp. 15) corresponds to 2.405GHz (resp. 2.480GHz), i.e. channel 15 (resp. 26) according to the IEEE802.15.4 numbering scheme. Bit 0 corresponds to channel 0. For continuous wave and continuous modulation tests, enable exactly one channel.
[in]powerTransmit power, in dB. Valid values are 0 and 8.
[in]stationIdDevice stationId
[in]numRepeatsNumber of times to repeat the packet sequence (0=repeat forever). Applies only to packet transmission tests.
[in]numSubtestsNumber of packets in each sequence. This parameter is only used for a packet test. Maximum allowed value is 10.
[in]subTestsPointer to an array of numSubtests sequence definitions (up to 10). This parameter is only used for packet test. Each entry contains:
  • the length of the packet (must be between 2 and 125 bytes).
  • the delay between this packet at the next one, in microseconds.
[out]rcLocation to write the return code to (details below).

This function calls the DN_API_LOC_CMD_TESTRADIOTX command of the local interface. There are in two elements which can be considered "return codes" when calling this function:

  • The value returned by this function merely indicates whether the command could be issued to the local interface.
  • The outcome of that call is written at the location pointed to by the rc parameter. Consult the testRadioTxExt section in the "SmartMesh IP Mote Serial API Guide" (http://www.linear.com/docs/41886); it lists the possible return codes and their meaning.
Returns
DN_ERR_NONE if the function completes successfully.
DN_ERR_ERROR if the function can not be completed successfully.

[in]sendtoPointer to the structure containing the packet and its metadata.
[in]lengthLength of the payload, in bytes.
[out]rcLocation to write the return code to (details below).

This function calls the DN_API_LOC_CMD_SENDTO command of the local interface. There are in two elements which can be considered "return codes" when calling this function:

  • The value returned by this function merely indicates whether the command could be issued to the local interface.
  • The outcome of that call is written at the location pointed to by the rc parameter. Consult the sendTo section in the "SmartMesh IP Mote Serial API Guide" (http://www.linear.com/docs/41886); it lists the possible return codes and their meaning.
Returns
DN_ERR_NONE if the function completes successfully.
DN_ERR_ERROR if the function can not be completed successfully.
